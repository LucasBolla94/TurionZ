# Plan 16-01: Self-Improvement Weekly Analysis

## Objective

Implement the full self-improvement cycle from specs/self-improvement.md.
Every Sunday at 3AM, TurionZ automatically analyzes the week's activity,
learns from mistakes, and verifies if last week's changes helped.

## Context

- PostgreSQL has conversations, messages, activity_logs tables
- OpenRouter provider exists for cheap model access
- specs/self-improvement.md defines 5-step cycle
- SchemaManager for on-demand table creation (lessons_learned, weekly_reports)

## Tasks

### Task 1: SelfImprover core service

**Files:** `src/core/SelfImprover.ts`, `src/types/index.ts`

1. Create `SelfImprover` singleton:

   - `scheduleWeeklyAnalysis()` — sets up timer for Sunday 3AM (configurable)
   - `runAnalysis()` — full 5-step cycle:

     **Step 1 — COLLECT:**
     - Query conversations from last 7 days
     - Query activity_logs from last 7 days
     - Query error entries from logs
     - Query previous week's applied changes (if any)

     **Step 2 — FRAGMENT:**
     - Estimate total token count (rough: chars / 4)
     - If > 50k tokens: split into ~20k chunks
     - Each chunk gets analyzed independently
     - If < 50k: analyze as single batch

     **Step 3 — ANALYZE (cheap model via OpenRouter):**
     - For each fragment, send to cheap model with analysis prompt:
       "What went well? What went wrong? What did the user correct?
        What patterns repeat? What improvements to suggest?"
     - Consolidate responses from all fragments
     - Extract categorized lessons: technical, preference, pattern, tool, communication

     **Step 4 — VERIFY PREVIOUS CHANGES:**
     - Load last week's applied_changes from lessons_learned
     - Compare metrics before vs after:
       - Error count this week vs last week
       - User corrections this week vs last week
     - Mark each change as: beneficial (keep), harmful (revert), neutral (keep one more week)
     - Revert harmful changes (update MEMORY.md, remove lesson)

     **Step 5 — APPLY:**
     - Save new lessons to lessons_learned table (on-demand creation)
     - Save weekly report to weekly_reports table (on-demand creation)
     - Update .agents/MEMORY.md with key lessons (append, max 20 entries)
     - Log everything via ActivityLogger

   - `getLastReport()` — returns most recent weekly report
   - `getLessons(category?)` — query lessons by category

2. Add types to `src/types/index.ts`:
   - `Lesson`, `WeeklyReport`, `ChangeVerification`

3. Model selection:
   - Use `ProviderFactory.createForSubAgent()` with cheapest available model
   - Default to 'meta-llama/llama-3.1-8b-instruct' or similar cheap model
   - Configurable via env var `ANALYSIS_MODEL`

### Task 2: Integration and scheduling

**Files:** `src/index.ts`, `src/core/AgentController.ts`

1. **src/index.ts**:
   - Initialize SelfImprover on startup
   - Call `scheduleWeeklyAnalysis()`
   - On shutdown: cancel scheduled timer

2. **AgentController**:
   - Add method `getWeeklyReport()` for Thor to share analysis results with user
   - Add method `getLessonsForContext()` — returns relevant lessons to inject into system prompt
   - In `buildSystemPrompt()`: append recent lessons as context hints

3. Scheduling mechanism:
   - Use `setInterval` with day-check (simple, no cron dependency)
   - Every hour, check: is it Sunday and between 3:00-3:05 AM and hasn't run this week?
   - If yes: run analysis
   - Track last run date in recovery_state to survive restarts

## Verification

- Weekly analysis runs and generates lessons
- Large conversation sets are properly fragmented
- Lessons saved to PostgreSQL with correct categories
- Previous changes verification marks beneficial/harmful correctly
- MEMORY.md updated with new lessons
- Weekly report saved with all metrics
- Scheduling survives restart (tracks last run date)
